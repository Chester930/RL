> ⚠️ **Q 值爆炸**：使用 real_ratio=0.05、lr=3e-4、sac_updates=10 的原始 CPU demo 配置，
> 在步數 ~28k 時發生嚴重數值爆炸。

| 步數（第一次）| 評估回報 | critic_loss | actor_loss | alpha |
|---|---|---|---|---|
| 10,000 | -1291.9 | — | — | — |
| 20,000 | -1258.5 | — | — | — |
| 28,000（中止）| — | **10,763,309,056** | **1,350,942** | **5,615** |

**根本原因**：
```
real_ratio=0.05 → 訓練批次中 95% 來自模型緩衝區
模型生成資料包含累積誤差 → SAC 的 Q 目標不斷被高估
高估 Q 值 → auto-alpha 推高溫度以控制熵 → alpha 5,615
alpha 爆炸 → 策略完全隨機化 → 更多 OOD 資料進入模型緩衝區
（正向回饋迴圈，無法自我修復）
```

## 為何 MBPO 在此環境難以收斂？

```
問題 1：小 real_buffer 的模型品質
  前 1k 步：random steps（1k 筆真實資料）
  前 10k 步：real_buffer 僅有 10k 筆，模型訓練不足
  模型品質差 → 生成的 model rollouts 有大誤差
  real_ratio=0.3 雖然比 0.05 好，但 70% 批次仍來自有誤差的模型

問題 2：SAC + 模型誤差的正向回饋
  模型誤差 → Q 目標偏高 → critic_loss 大 → 梯度更新不穩定
  不穩定更新 → alpha 升高 → 策略更隨機 → 探索資料多樣但噪聲大

問題 3：Pendulum 的非線性動態
  Pendulum 的鐘擺動態對模型誤差很敏感，1 步預測誤差會在 rollout 中累積

建議解法：
  (A) 增加 rollout_length 排程（從 1 逐步增加到 5）
  (B) 使用更大 ensemble（7 個成員，論文設定）
  (C) 改用 Ant-v4 / HalfCheetah 等環境（論文主要測試環境，動態更平滑）
  (D) GPU 上跑 100k-300k 步完整訓練
```

## MBPO 核心原理

```
每個真實步數的訓練迴圈（MBPO 核心）：
  1. 收集 1 個真實轉換（s, a, r, s'）→ 存入 real_buffer
  2. 每 250 步：在 real_buffer 上訓練整合動態模型 p_θ(s'|s,a)
  3. 從 real_buffer 取樣狀態 → 用模型展開 k 步 → 存入 model_buffer
  4. 混合批次（30% real + 70% model）→ 5 次 SAC 梯度更新

核心優勢（相對純 Model-Free SAC）：
  model rollouts 等效於「免費的合成資料」
  → 用少量真實互動訓練高效策略
  → 論文報告樣本效率提升 5-40×（Hopper/Ant 環境上）

核心挑戰：
  模型誤差（Model Bias）會累積進 Q 估計
  → MBPO 使用短 rollout（k=1）+ 整合不確定性估計加以緩解
```
