# CPO 訓練日誌

> 環境：SafePendulum（Pendulum-v1 + 代價信號）  
> 安全約束：|θ̇| > 2.0 rad/s → cost=1，每集限 **5 步**違規  
> 訓練規模：600K 步，n_steps_per_update=4096（146 次更新）  
> 裝置：CPU（PyTorch 2.11.0+cpu）

---

## 演算法特有指標

| 指標 | 說明 |
|---|---|
| `eval/mean_reward` | 評估時每集平均獎勵（越高越好） |
| `eval/mean_cost` | 評估時每集平均違規步數（目標 ≤ 5） |
| `train/kl` | 每次更新的實際 KL 散度（應 ≤ max_kl=0.01） |
| `train/mean_ep_cost` | 訓練中每集違規步數 |
| `train/update_type` | 0=TRPO 步（安全時）、1=CPO 步（違規時） |

## 更新類型分佈

```
CPO 步（違規，代價 > 5）：100% 的更新（146/146）
TRPO 步（安全，代價 ≤ 5）：0% 的更新
```

**解讀**：整個訓練過程代價始終超過 5.0，因此 CPO 每次都啟用約束更新模式（硬約束自然梯度）。這展示了 CPO 在嚴苛約束下持續工作的機制。

## 觀察與分析

**早期行為**：初始策略代價高達 108/集（200 步中有 108 步違規），遠超限制 5。CPO 從第一次更新就進入約束模式。

**CPO 步的觸發條件**：每次更新的 mean_ep_cost > 5.0，因此全程都是 CPO 步。這說明在嚴苛約束（cost_limit=5）下，代理人沒有「安全預算」可以浪費。

**KL 散度穩定性**：KL 在 0.003–0.009 之間，始終 ≤ max_kl=0.01，自然梯度步長控制有效。

**CPU 限制**：代價最低降至 6.5（差 1.5 步達標），在更多訓練步數或 GPU 加速下可能收斂。每 600K 步耗時約 12 分鐘（CPU）。

## 對比：CPO vs PPO-Lagrangian

| 項目 | CPO | PPO-Lagrangian |
|---|---|---|
| 更新機制 | 自然梯度 + 硬約束（信任域） | 梯度剪裁 + λ 軟約束 |
| eval 代價最低 | **6.5**/集 | 8.9/集 |
| 最終 eval 獎勵 | -1483.7 | -1817.7 |
| λ / 約束強制 | 無（硬約束）| λ 從 0→5.68 |
| 更新全為安全模式？ | 否（全 CPO，始終違規） | 否（λ 持續上升） |
| 訓練速度（600K）| ~12 min CPU | ~6 min CPU |
| 每步計算量 | 高（共軛梯度 + 線搜尋） | 低（PPO 標準更新） |
