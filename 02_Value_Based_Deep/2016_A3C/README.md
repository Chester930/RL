# A3C — 非同步優勢演員-評論家 (Asynchronous Advantage Actor-Critic)

## 論文

Mnih, V., Badia, A. P., Mirza, M., Graves, A., Lillicrap, T., Harley, T., Silver, D., & Kavukcuoglu, K. (2016).  
*Asynchronous Methods for Deep Reinforcement Learning*. ICML 2016. arXiv:1602.01783.

---

## 核心思想 (Key Idea)

A3C 維護一個**全域網路 (Global Network)**，由多個並行執行的**工作者程式 (Worker Processes)** 進行更新，每個工作者都有自己的環境副本。工作者在本地計算梯度並非同步地應用到全域網路中，這提供了天然的探索多樣性，並可作為經驗回放 (Experience Replay) 的快速替代方案。

**白話解釋：**

A3C 和 DQN 家族有一個根本上的不同——它同時用了「演員」和「評論家」兩個角色：

```
DQN：只有一個 Q 網路，學「每個動作值多少分」
A3C：演員學「在這個狀態下，每個動作應該有多大機率」
     評論家學「這個狀態本身值多少分」
```

為什麼要兩個？

- **演員（Actor）**：告訴你「該做什麼」，但它自己沒辦法判斷「做得好不好」
- **評論家（Critic）**：看著演員表演，告訴它「這次比平均好還是差」

兩者互相配合：評論家給演員打分，演員根據評分調整策略。

**A3C 的「A」代表什麼：**

- **Asynchronous（非同步）**：多個工作者同時跑，不互相等待
- **Advantage（優勢）**：用「比預期好多少」來衡量動作好壞（詳見損失函式）
- **Actor-Critic（演員-評論家）**：網路同時輸出策略和狀態價值

## 架構 (Architecture)

```
全域網路 (Global Network, 共享權重)
    |
    ├── 工作者 1 (環境副本 1) → 收集 n 步 → 計算梯度 → 應用至全域
    ├── 工作者 2 (環境副本 2) → 收集 n 步 → 計算梯度 → 應用至全域
    ├── 工作者 3 (環境副本 3) → 收集 n 步 → 計算梯度 → 應用至全域
    └── 工作者 N (環境副本 N) → 收集 n 步 → 計算梯度 → 應用至全域
```

每個工作者：複製全域權重 → 執行 n 步 → 累積梯度 → 更新全域

**白話解釋：**

想像一家大型補習班在同時幫學生批改作業：

```
全域網路   = 補習班的「標準答案本」（所有老師共用）

工作者 1  = 老師A，帶著自己的一班學生（環境副本1）去練習
工作者 2  = 老師B，帶著另一班學生（環境副本2）去練習
工作者 N  = 老師N，同上...

每個老師：
  1. 先影印一份「標準答案本」帶走
  2. 和學生練習 n 步，記錄哪裡錯了（計算梯度）
  3. 把「這樣修改答案本比較好」的建議交回補習班
  4. 補習班收到建議就立刻更新答案本（不等其他老師）
```

關鍵：各老師「非同步」回報——老師A剛交完報告，老師B還在第3步，老師C才剛開始。不互相等待，效率高。

**網路架構（共享骨幹，兩個輸出頭）：**

```
輸入狀態（4 維）
  │
  ▼
共享骨幹（MLP 256維）← 提取對演員和評論家都有用的特徵
  │
  ├──▶ 演員輸出頭 → logits → softmax → π(←), π(→)
  │     「推左的機率 40%，推右的機率 60%」
  │
  └──▶ 評論家輸出頭 → V(s)
        「這個狀態我預期能再活 87 步」
```

## 損失函式 (Loss Function)

```
優勢 (Advantage):  A_t = G_t - V(s_t)    其中 G_t = r_t + gamma*r_{t+1} + ... + gamma^{n-1}*r_{t+n} + gamma^n * V(s_{t+n})

演員損失 (Actor loss):   L_actor  = -log pi(a_t | s_t) * A_t
評論家損失 (Critic loss):  L_critic = (V(s_t) - G_t)^2
熵損失 (Entropy):         L_entropy = -H(pi(s_t))   [鼓勵探索]

總損失:  L = L_actor + c_v * L_critic - c_e * L_entropy
```

**白話解釋（逐項拆解）：**

---

**① 優勢 A_t：「這次比預期好多少？」**

```
G_t   = 從這步開始，接下來 n 步實際拿到的折扣回報（真實成績）
V(s_t) = 評論家對「這個狀態」的預期（平均成績）
A_t   = G_t - V(s_t)   ← 實際 - 預期
```

具體例子：
```
評論家說：「從這個狀態出發，平均能再活 80 步」（V = 80）
這次實際撐了 120 步（G = 120）
優勢 A = 120 - 80 = +40   ← 比預期好！

評論家說：「平均能活 80 步」
這次只活了 30 步（G = 30）
優勢 A = 30 - 80 = -50    ← 比預期差
```

類比：你這學期的物理平均是 70 分（V），這次小考考了 85 分（G），優勢 = +15（超出預期）。

---

**② 演員損失：「根據這次的表現，調整選動作的機率」**

```
L_actor = -log π(a|s) × A_t
```

分兩種情況：

```
A_t > 0（這次比預期好）：
  → 我們想「增加」選這個動作的機率
  → log π 越大（機率越高）越好
  → 損失 = -log π × (+) = 負數，梯度下降會提高 π

A_t < 0（這次比預期差）：
  → 我們想「降低」選這個動作的機率
  → 損失 = -log π × (-) = 正數，梯度下降會降低 π
```

用人話說：**「如果這次比平均好，下次再遇到同樣情況就多選這個動作；比平均差，就少選。」**

注意：不是「好就加、壞就減」的死規則，而是「比預期好才加、比預期差才減」。這是「優勢」最重要的地方——避免對所有好的結果都盲目加機率。

---

**③ 評論家損失：「讓 V(s) 的預測更準」**

```
L_critic = (V(s_t) - G_t)²
```

就是標準的 MSE：評論家預測的 V(s) 和實際發生的 G_t 之差的平方。訓練目標是讓評論家越來越準——預測準了，優勢 A_t 才有意義。

類比：如果評論家的「平均成績」估計一直不準（不知道真實水平在哪），演員就不知道「這次到底算好還是算差」，整個系統就亂了。

---

**④ 熵獎勵：「保持好奇心，不要太早認定一個答案」**

```
H(π) = -Σ π(a|s) × log π(a|s)   ← 策略的「不確定程度」

策略接近均勻分佈時（π ≈ [0.5, 0.5]）：H 高（很不確定）
策略接近確定性時  （π ≈ [1.0, 0.0]）：H 低（非常確定）
```

我們在損失裡**減去**熵（L = ... - c_e × H），讓最小化損失等於最大化熵。也就是說，我們在獎勵網路「保持不確定、多探索」。

為什麼要這樣做？

```
沒有熵獎勵：訓練初期網路很快就 100% 選某個動作
           → 再也不探索其他可能 → 卡在次優策略

有熵獎勵：網路被迫保持一定的隨機性
          → 持續探索 → 更有機會找到真正好的策略
```

類比：考試選擇題，如果你一直都只選A，你永遠不會知道B有時候才是對的。熵獎勵就是在說「不要太早下定論，多試幾個選項」。

## 非同步 vs 同步 (A2C)

| 特性 | A3C (非同步) | A2C (同步) |
|----------|------------|-----------|
| 工作者 | 非同步、獨立執行 | 同步、等待所有人完成 |
| 探索多樣性 | 豐富 (不同策略) | 單一 |
| GPU 效率 | 較差 (小批次) | 較佳 (大批次) |
| 實作難度 | 複雜 | 簡單 |
| 效能表現 | 經驗上相近 | 實務上通常較佳 |

A2C (同步 A3C) 在實務上通常能達到或超越 A3C 的表現，且實現更簡單。

**白話解釋：**

```
A3C（非同步）：老師們各自做完就交，不互相等待
  優點：快，某個工作者跑很慢不會拖累其他人
  缺點：更新時用的是「舊版答案本」算出來的梯度（有延遲偏差）

A2C（同步）：所有老師同一批做完，一起交
  優點：用的都是最新版答案本，梯度估計更準確
  缺點：最慢的老師決定整體速度
```

本資料夾的實作是**單執行緒版本（等價於 A2C）**——因為要真正實現 A3C 的非同步多程序需要相對複雜的程式結構（`torch.multiprocessing`），但核心數學完全相同。

## 為什麼不用經驗回放? (Why No Replay Buffer?)

A3C 使用多個並行工作者而非經驗回放，是為了：
- 解除訓練資料的相關性 (不同的工作者處於不同的狀態)。
- 實現線上更新 (無儲存開銷)。
- 隨 CPU 核心數自然擴充套件。

**白話解釋：**

DQN 需要 Replay Buffer 是因為：連續幾步的資料高度相關（今天的狀態是昨天動作的直接結果），用這種資料訓練很不穩定。

A3C 的解法不同：

```
DQN 的解法：把過去的資料存起來，隨機抽取 → 打亂相關性
A3C 的解法：同時跑多個環境 → 不同工作者在完全不同的情況 → 天然不相關
```

類比：
```
DQN：一個人玩遊戲，把每局錄影存起來，事後隨機剪片段來複習
A3C：同時有 8 個人在玩，他們各自在不同進度、不同局面
     直接把 8 個人當下的經驗拿來一起學，自然就不相關了
```

代價是：A3C 是**線上策略（On-policy）**——只能用當前策略收集的資料來更新。DQN 是**離線策略（Off-policy）**——舊資料也能用。A3C 不能重複使用過去的資料，每步資料只用一次就丟。

## 實作注意 (Implementation Note)

完整的非同步實作需要：
```python
global_net.share_memory()
optimizer = SharedAdam(global_net.parameters(), lr=7e-4)
workers = [mp.Process(target=worker_fn, args=(global_net, optimizer, ...))
           for _ in range(n_workers)]
```

**白話解釋：**

- `share_memory()`：讓全域網路的引數儲存在可以跨程序讀寫的記憶體區塊
- `SharedAdam`：普通 Adam 最佳化器不能跨程序使用，需要特殊版本
- `mp.Process`：為每個工作者開一個獨立的作業系統程序（不是執行緒）

為什麼用程序不用執行緒？Python 的 GIL（全域直譯器鎖）讓多執行緒無法真正平行計算——用程序才能讓多個 CPU 核心真正同時跑。

## A3C vs DQN 家族：根本差異

| 面向 | DQN 家族 | A3C |
|:---|:---|:---|
| 策略型別 | 確定性（ε-greedy，幾乎都選最好的）| 隨機（用機率分佈選動作）|
| 探索方式 | ε 從 1.0 衰減到 0.01 | 熵獎勵（自動調整探索力度）|
| 輸出 | Q(s, a)：每個動作的分數 | π(a\|s)：每個動作的機率 + V(s) |
| 資料重用 | Replay Buffer 可重複利用 | 每筆資料只用一次就丟 |
| 更新時機 | 每步都更新 | 每 n 步（rollout 結束後）更新一次 |
| 需要目標網路？| 是（穩定 TD 目標）| 否（n 步回報本身就較穩定）|
