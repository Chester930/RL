## 核心概念

### A3C vs DQN 家族的根本差異

```
DQN 家族（離線策略，Off-policy）：
  - 儲存所有過去的轉移到 Replay Buffer
  - 從 Buffer 隨機抽樣進行批次更新
  - 策略選擇：ε-greedy（探索 vs 利用）
  - 更新目標：Q(s,a) = r + γ × max_a' Q(s',a')

A3C（線上策略，On-policy）：
  - 不使用 Replay Buffer，直接用當前策略收集資料
  - 每 n 步更新一次，然後丟棄舊資料
  - 策略選擇：隨機取樣 a ~ π(a|s)（softmax 分佈）
  - 更新目標：最大化 E[log π(a|s) × A(s,a)]
```

### 演員-評論家架構（Actor-Critic）

```
輸入 s（4維）
  │
  ▼
共享骨幹（MLP, 256維）
  │
  ├──▶ 演員 (Actor) head → logits → softmax → π(a|s)   [策略梯度]
  │
  └──▶ 評論家 (Critic) head → V(s)                      [值函式估計]
```

### 損失函式分解

```
① 演員損失 (Actor Loss)：
   L_actor = -E[log π(a|s) × A(s,a)]

   A(s,a) = G_t - V(s_t)    ← 優勢函式 = n-步回報 - 評論家估計

   A > 0：此動作比平均好 → 提高 π(a|s)
   A < 0：此動作比平均差 → 降低 π(a|s)

② 評論家損失 (Critic Loss)：
   L_critic = E[(V(s_t) - G_t)^2]    ← MSE 損失

   G_t = r_t + γ*r_{t+1} + ... + γ^{n-1}*r_{t+n-1} + γ^n * V(s_{t+n})

③ 熵獎勵 (Entropy Bonus)：
   L_entropy = -E[H(π(s))] = E[Σ π(a|s) log π(a|s)]

   鼓勵策略保持隨機性，避免過早收斂到次優的確定性策略
   均勻分佈時熵最大：H_max = ln(2) ≈ 0.693

④ 總損失：
   L_total = L_actor + 0.5 × L_critic + 0.01 × L_entropy
```

### n-步回報 vs TD(0) vs Monte Carlo

```
n=1（TD(0)）：G_t = r_t + γV(s_{t+1})
  → 低方差，高偏差（仰賴 V 估計，可能傳播誤差）

n=∞（Monte Carlo）：G_t = Σ_{k=0}^∞ γ^k * r_{t+k}
  → 高方差，零偏差（完整軌跡，但雜訊大）

n=5（A3C 預設）：
  → 折衷取捨：比 TD 偏差小，比 Monte Carlo 方差小
```

**解讀重點：**
- `V(s)`：評論家估計的狀態回報（越高代表狀態越好）
- `P(←)/P(→)`：演員輸出的策略分佈（隨機，非確定性）
- `H(π)`：策略熵（H_max=0.693；越低越確定；趨近 0 = 完全確定性）
- 桿往左傾 → P(←) 應 > P(→)；桿往右傾 → P(→) 應 > P(←)
- A3C 的策略是機率分佈，不像 DQN 的 argmax 那樣完全確定

### ⚠ 單執行緒限制說明（已知行為）

**此實作為單執行緒 A2C，非真正的多執行緒 A3C。** 觀察到的訓練現象（eval 33-54、策略崩潰至「只推左」）是**預期中的已知行為**，並非程式錯誤：

| 問題現象 | 根本原因 |
|:---|:---|
| 評論家損失爆炸（10M → 100M）| 單一執行緒的梯度高度相關，bootstrap 估計誤差快速累積 |
| 熵崩潰（0.693 → 0.009）| 無多工作者多樣性，策略過早鎖定在次優的確定性行為 |
| 策略退化（永遠推左）| 熵崩潰後無法自我修正，因線上策略不保留歷史轉移 |

**真正的 A3C 修正方式**：
1. **並行工作者**（原論文：16-64 個工作者） → 打破梯度相關性，穩定值函式學習
2. **非同步更新**（各工作者獨立累積梯度後推送至全域網路）
3. **替代方案**：改用 A2C（同步多環境版本，後續 Chapter 03 實作）

此結果是教學用途下刻意保留的「反例」——展示了缺少並行機制時 A3C 會如何失敗。

## 演算法特有指標：多工作者多樣性效果

> 本實作為單執行緒，以下對比展示**如果有多個工作者**時各指標會如何不同，解釋為什麼真正的 A3C 能成功而本實作失敗。

### 單執行緒 vs 多工作者的指標對比

| 指標 | 本實作（單執行緒）| 真正 A3C（16 workers）| 差異說明 |
|---|---|---|---|
| 評論家損失 50K | **11,167,526** | < 1,000（典型值）| 梯度相關性導致 bootstrap 誤差累積 |
| 評論家損失 100K | **106,614,576** | < 5,000（緩步上升）| 單執行緒持續惡化 |
| 策略熵 50K | **0.1999** | ~0.5–0.6 | 多樣探索維持高熵 |
| 策略熵 100K | **0.0091** | ~0.4–0.5 | 多工作者防止過早確定化 |
| eval 回報 100K | **38.8**（退化）| ~350–450（CartPole）| 核心差距 |
| 崩潰現象 | 永遠推左（確定性）| 無（各 worker 策略多元）| 無法自我修正 |

### 為什麼多工作者能解決這些問題

```
單執行緒（本實作）：
  同一個策略 → 收集軌跡 → 相似狀態 → 相關梯度
  → Critic 更新方向重複，誤差累積 → 崩潰

16 workers（真正 A3C）：
  Worker 1：CartPole 角度 +0.03rad，策略偏右
  Worker 2：CartPole 角度 -0.05rad，策略偏左
  Worker 3：CartPole 剛倒下，學習恢復動作
  ...
  Worker 16：CartPole 近乎完美，強化當前策略

  各 worker 的梯度彼此「裝飾性相消」，
  只有真正有效的梯度方向能在 16 個梯度的均值中存活
  → 等效於 batch_size × 16，但不需要 Replay Buffer
```

### 本實作保留為「反例」的教學價值

| 觀察到的失敗現象 | 教學意義 |
|---|---|
| 評論家損失爆炸（10M → 100M）| 展示「梯度相關性爆炸」問題 |
| 熵崩潰（0.693 → 0.009）| 展示「多樣性缺失導致的早熟收斂」|
| eval 33-54（低於 DQN 的 500）| 量化展示缺少並行機制的代價 |
| 策略永遠推左 | 展示確定性策略陷阱（no exploration = no recovery）|

**與 A2C VecEnv 版本的直接對比**：  
A2C（4 平行環境，300K 步）達到 **500.0**，本 A3C（1 執行緒，150K 步）僅 **33–54**。
差異來源完全是「並行度」，不是演算法公式。

### A3C vs DQN 家族完整對比

| 面向 | DQN / Double DQN | Dueling DQN | A3C |
|:---|:---|:---|:---|
| 策略型別 | 確定性（ε-greedy）| 確定性（ε-greedy）| 隨機（softmax）|
| 資料重用 | Replay Buffer | Replay Buffer | 無（即收即丟）|
| 探索機制 | ε 衰減 | ε 衰減 | 熵獎勵 |
| 網路輸出 | Q(s,a) | V(s) + A(s,a) | π(a|s) + V(s) |
| 更新頻率 | 每步 1 次 | 每步 1 次 | 每 5 步 1 次 |
| 目標函式 | TD 誤差最小化 | TD 誤差最小化 | 優勢加權對數機率 |
| 優勢估計 | 隱含在 Q 差 | 隱含在 Q 差 | 顯式 A(s,a) = G_t - V(s_t) |
