- 演算法完整性：GAE + 優勢歸一化 + entropy bonus + gradient clipping 全部正確實作

## CartPole-v1（2026-05-24，VecEnv 重跑）

**動機**：第一次訓練（單環境）僅達 357.5 且不穩定，
改用 4 個平行環境（SyncVectorEnv）以降低梯度估計的方差，
重現 A2C 論文中多工作者平行化的效果。

**程式修改**：
- `agent.py`：新增 `select_actions(states_batch)` 支援批次動作選擇；
  `update()` 改為接受 (T, N) 形狀的陣列，展平後計算 GAE 再更新
- `train.py`：改用 `gym.vector.SyncVectorEnv`（4 envs），
  n_episodes → n_updates（total_steps // (n_envs × n_steps)）

| 引數 | 數值 |
|------|------|
| n_envs | **4**（新增）|
| total_steps | 200,000 |
| lr | 7e-4（RMSprop）|
| n_steps | 5 |
| gae_lambda | 0.95 |
| c_v | 0.5 |
| c_e | 0.01 |
| 有效批次大小 | n_envs × n_steps = **20** |
| 總更新次數 | 10,000 |

**訓練時長**：2026-05-24（< 5 分鐘，CPU）

### 觀察

```
VecEnv 並未改善 A2C 的根本不穩定性：

問題根源：A2C 沒有「信賴域」保護
  每次更新可能踏出過大的策略步伐
  → 策略破壞後，後續資料品質也下降
  → 強化了崩潰（與 MBPO 的 Q 爆炸類似）

VecEnv 的實際效果：
  ✅ 提供更多樣的梯度（4 個不相關的環境序列）
  ✅ 收斂速度更快（90k 步達 117，單環境需 ~180k 回合才接近）
  ❌ 批次大小增大 4x → 等效學習率提高 → 更新步伐更大 → 崩潰更劇烈

真正的解法：PPO（已完成，CartPole 500 滿分）
  PPO 的 clip(ratio, 1-ε, 1+ε) 限制單步策略更新幅度
  → 即使用 VecEnv，更新也不會踏出致命的一步
```

## CartPole-v1（2026-05-24，VecEnv 第二次，調參版）

**動機**：第一次 VecEnv 跑法（lr=7e-4, n_steps=5, c_e=0.01）峰值 260.7 後仍崩潰至 14.8。
分析根因：批次大小從 5→40（4×8），等效 LR 過高，加上 rollout 太短導致 GAE 信號弱，
熵係數不足讓策略過早收斂。針對三點修正：

| 引數 | 第一次 VecEnv | **第二次（修正）** | 原因 |
|------|---|---|---|
| lr | 7e-4 | **2e-4** | 批次大 4× 需縮小 LR |
| n_steps | 5 | **10** | 更長 rollout 讓 GAE 傳遞更多步 |
| c_e | 0.01 | **0.02** | 維持探索防止策略過早低熵崩潰 |
| total_steps | 200,000 | **300,000** | 給調整後配置更多學習時間 |
| 有效批次 | 4×5 = 20 | 4×10 = **40** | — |
| 總更新次數 | 10,000 | 7,500 | — |

**訓練時長**：2026-05-24（約 10 分鐘，CPU）

### 觀察

```
調參效果分析：

lr 2e-4（↓ 3.5×）：最關鍵的改動
  更小的策略更新步伐 → 峰值後不再一步踏入崩潰區
  代價：學習更慢（前 40k 步仍在 10-21 低位）

n_steps 10（↑ 2×）：改善 GAE 信號
  更長的 rollout 讓「倒下 = 壞」的信號傳回更多步
  幫助評論家更準確估計長期回報

c_e 0.02（↑ 2×）：維持探索
  更高的熵懲罰防止策略過早變得確定性
  讓策略在 200-220k 的短暫回落後仍能回升至 500

最終結果：300k 步達到 500 ± 0（10 集全部滿分）
雖然仍有波動（非 PPO 那樣穩定的單調上升），
但最終收斂，達成演算法完整性驗證目標。
```
