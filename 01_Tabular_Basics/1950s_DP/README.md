## 1. 它要解決什麼問題？

想像你在玩一個格子地圖的遊戲，目標是從起點走到終點。

你想知道的是：**在每個格子上，我應該往哪個方向走，才能最快、最安全地到達終點？**

動態規劃（DP）就是用來回答這個問題的方法。

但它有一個前提：**你必須事先知道地圖的所有規則。**

包括：
- 從每個格子往某個方向走，會去到哪裡
- 每種移動能得到多少分
- 哪些格子是陷阱、哪個是終點

這些規則在程式裡叫做 `env.P[s][a]`，可以直接查表。知道了這張表，DP 就能精確算出最佳走法，完全不需要實際去試。

## 2. 測試環境：FrozenLake

### 地圖說明

我們用 `FrozenLake-v1` 這個遊戲來測試，它是一個 4×4 的格子地圖：

| (列/行) | 0 | 1 | 2 | 3 | 備註 |
| :---: | :---: | :---: | :---: | :---: | :--- |
| **0** | **S** | F | F | F | **S** = 起點 (Start) |
| **1** | F | **H** | F | **H** | **F** = 普通格子，可以走 (Frozen) |
| **2** | F | F | F | **H** | **H** = 洞口，掉下去遊戲結束 (Hole) |
| **3** | **H** | F | F | **G** | **G** = 終點，抵達得 1 分 (Goal) |

- 共 16 個格子（狀態）
- 每個格子可以選 4 個方向：0=左、1=下、2=右、3=上

### 關鍵特性：滑溜冰面

這個地圖的冰面是滑的，你選擇的方向不一定是實際移動的方向：

- 選「往下」→ 實際上有 **1/3 機率往下**、1/3 往左、1/3 往右
- 你沒辦法完全控制自己走哪裡

這表示就算有最完美的策略，也不可能每次都成功。這就是現實世界中「不確定性」的模擬。

### 環境提供的資訊

```
n_states  = 16      # 16 個格子
n_actions = 4       # 4 個方向
env.P[s][a]         # 規則表：P[格子][方向] = [(機率, 下一格, 獎勵, 是否結束), ...]
```

## 3. 核心基礎：貝爾曼方程式

在說演演算法之前，要先搞清楚兩個基本概念。

### 概念一：V(s)，每個格子值多少分

**V(s)** 就是：站在格子 s 上，如果從這裡繼續走下去，我**預期能得到幾分**。

- 靠近終點的格子 → V 高（離目標近，拿到分數的機率大）
- 靠近洞口的格子 → V 低（容易掉下去，拿不到分）
- 洞口本身 → V = 0（掉下去就結束了，什麼都拿不到）

### 概念二：γ（gamma），遠的獎勵要打折

現在得到 1 分，比 10 步後才得到 1 分更有價值。

```
γ = 0.99 表示：
  1 步後的獎勵，只值現在的 0.99 倍
  2 步後的獎勵，只值現在的 0.99² ≈ 0.98 倍
  10 步後的獎勵，只值現在的 0.99^10 ≈ 0.90 倍
```

越晚拿到的獎勵，換算成現在的價值就越少。

### 貝爾曼方程式：一個格子的分數怎麼算

一個格子的 V，可以用它**鄰居的 V** 來計算：

```
這個格子的 V = 走出去能得到的獎勵 + (下一個格子的 V × 折扣 γ)
```

這是自我參照的：算 A 格的分數，需要知道 B 格的分數；算 B 格又需要 C 格……

DP 的做法就是：**先全部設為 0，然後反覆套用這個公式更新，直到所有格子的分數不再變化。**

兩種方程式的差別在於：

- **策略評估用**（你已經有一套走法）：只算你規定的那個方向的期望分數
- **最佳化用**（你想找最好走法）：算所有方向，取最高的那個

```
策略評估：V(s) = 用 policy[s] 那個方向走，期望能得幾分
最佳化：  V(s) = 試所有方向，取最高分的那個
```

## 4. 演演算法一：價值疊代 (Value Iteration)

### 白話說明

> 先把每個格子的分數標出來，最後再根據分數決定走法。

**第一步：所有格子的分數設為 0（完全不知道）**

**第二步：對每個格子，試 4 個方向，看看哪個方向的期望分數最高，更新這個格子的分數**

```
格子 (3,1) 試 4 個方向：
  往左 → 可能到 (3,0) 那是洞口，分數 = 0 → 期望分數低
  往下 → 超出地圖，原地不動              → 期望分數中等
  往右 → 可能到 (3,2) 離終點近，分數高   → 期望分數最高 ✓
  往上 → 到 (2,1)，分數中等             → 期望分數中等

→ 更新 V(3,1) = 往右方向的期望分數
```

**第三步：重複第二步，直到所有格子的分數不再改變**

**第四步：分數穩定了，再看每個格子「往哪個方向分數最高」→ 那就是最佳走法**

### 重要觀念：VI 過程中沒有「走法」

整個過程只有一張「分數表」V(s)，從頭到尾都沒有記錄「應該走哪個方向」。

走法是**最後才推導**的：分數穩定後，對每個格子問「哪個方向的期望分數最高」，答案就是走法。

### 為什麼需要 324 次才收斂？

一開始所有格子都是 0 分，只有終點旁邊的格子在第 1 輪能算出非零值（因為它們一步就能到終點得 1 分）。

資訊從終點往外擴散，一格一格傳遞，每次都被 γ 折扣一點點，直到傳到起點為止。格子越遠，需要越多輪才能得到正確的分數值。

### 逐步運算記錄

執行 `python train.py` 後，同目錄會自動產生 `training_log.md`。

裡面記錄 VI 每個關鍵輪次（第 1、2、3 輪，之後每 50 輪，以及最終收斂輪）的完整資料：每輪的 δ 值、4×4 V 值表格、哪些格子被更新及變化量。

**規律：** 每輪「有值的區域」都往外多擴一圈，同時舊的值繼續被修正得更準確。最終需要 324 輪才能讓整張地圖的值完全收斂。

### 關鍵程式碼（agent.py）

**計算每個方向的期望分數 Q(s, a)：**

```python
# agent.py: _compute_q_values()
def _compute_q_values(self, s: int, P: dict) -> np.ndarray:
    q_values = np.zeros(self.n_actions)
    for a in range(self.n_actions):
        for prob, s_next, r, done in P[s][a]:
            q_values[a] += prob * (r + self.gamma * self.V[s_next] * (1 - done))
    return q_values
```

`(1 - done)` 的作用：如果下一格是洞口或終點（遊戲結束），未來就沒有更多分數，所以不加。

**價值疊代主迴圈：**

```python
# agent.py: value_iteration()
while True:
    delta = 0.0
    for s in range(self.n_states):
        v_old = self.V[s]
        q_values = self._compute_q_values(s, env.P)
        self.V[s] = np.max(q_values)          # 取 4 個方向裡最高的
        delta = max(delta, abs(v_old - self.V[s]))
    n_sweeps += 1
    if delta < self.theta:                    # 變化量夠小，代表收斂了
        break
```

## 5. 演演算法二：策略疊代 (Policy Iteration)

### 白話說明

> 先定一套走法，評估這套走法能得幾分，再根據分數改進走法，重複直到走法不再改變。

**第一步：隨便定一套初始走法（例如：所有格子都往左走）**

```
初始走法（policy）：
← ← ← ←
← ← ← ←
← ← ← ←
← ← ← ←
全部都往左，這是個很爛的走法，但沒關係
```

**第二步（評估）：按照這套走法走，計算每個格子的期望分數**

注意：這裡不試其他方向，只算「走法規定的那個方向」會得幾分，一直算到分數穩定為止。

「全往左」是爛走法，所以算出來的分數很低，但這是如實反映這套走法的結果。

**第三步（改進）：拿著算好的分數，對每個格子問「換個方向會更好嗎？」**

```
格子 (3,1) 目前走法 = 左：
  重新試 4 個方向，往右的期望分數比往左高
  → 把 policy[3,1] 改成「右」
```

**第四步：改完走法之後，回到第二步重新評估新走法**

**直到某輪改進後，所有格子的走法都沒有改變 → 這套走法已經是最佳的了，停止**

### 為什麼只需要 7 輪？

每輪的改進都是「有完整資訊支撐」的大幅更新——先把當前走法徹底評估清楚，再一口氣把所有格子能改的都改掉。相比 VI 的每次只微調一點點，PI 每輪改進的幅度大很多，所以輪數少得多。

### 逐步運算記錄

執行 `python train.py` 後，同目錄會自動產生 `training_log.md`。

裡面記錄 PI 每一輪（共 7 輪）的完整資料：評估後的 V^π 表格、哪些格子改變了方向、改變前後的箭頭、以及改進後的完整策略表格。

**規律：** 每輪評估都比上一輪更精確（因為走法更好了），改進時能做出更大的調整。PI 只需要 7 輪就能找到最佳走法，因為每輪都是「充分評估後的有資訊改進」，而不是像 VI 那樣每次只微調一點點。

### 關鍵程式碼（agent.py）

**步驟二：評估當前走法（只算固定方向的分數）：**

```python
# agent.py: _policy_evaluation()
while True:
    delta = 0.0
    for s in range(self.n_states):
        v_old = self.V[s]
        a = self.policy[s]             # 只用走法規定的那個方向
        self.V[s] = sum(
            prob * (r + self.gamma * self.V[s_next] * (1 - done))
            for prob, s_next, r, done in P[s][a]
        )
        delta = max(delta, abs(v_old - self.V[s]))
    if delta < self.theta:
        break
```

注意：這裡不取 max，只算 `policy[s]` 那個固定方向的期望分數。

**步驟三：改進走法（換成分數更高的方向）：**

```python
# agent.py: policy_iteration()
for s in range(self.n_states):
    old_action = self.policy[s]
    q_values = self._compute_q_values(s, env.P)
    self.policy[s] = np.argmax(q_values)        # 換成 4 個方向裡最好的
    if old_action != self.policy[s]:
        policy_stable = False
```

### 為什麼每次改進都不會變差？

因為我們是「知道了現在走法的分數之後，才換成更好的方向」。換的條件是「新方向分數 ≥ 舊方向分數」，所以每次只會持平或變好，絕對不會變差。

有限的格子數量 × 每次只進不退 → 一定會在有限輪數內找到最佳走法。

## 7. 結果解讀

### 每個格子的分數（V*）

把分數標在地圖上：

| (列/行) | 0 | 1 | 2 | 3 |
| :---: | :---: | :---: | :---: | :---: |
| **0** | S (0.542) | F (0.499) | F (0.471) | F (0.457) |
| **1** | F (0.558) | **H (0.000)** | F (0.358) | **H (0.000)** |
| **2** | F (0.592) | F (0.643) | F (0.615) | **H (0.000)** |
| **3** | **H (0.000)** | F (0.742) | F (0.863) | **G (0.000)** |

**洞口（H）= 0 分**：掉下去就結束，後面什麼都得不到。

**終點（G）= 0 分**：這不是說終點沒有價值，而是 V(s) 代表「站在這個格子之後還能得幾分」。到了終點遊戲就結束了，後面沒有步驟可以走，所以 V = 0。分數是在「走進終點那一步」拿到的，反映在終點旁邊的高分上（如 0.863）。

**終點左方 (3,2) = 0.863，最高分**：離終點最近，一步就有很高機率進去拿到 1 分。

**左側那一列分數較高**：雖然離終點遠，但遠離洞口、安全路線多，期望能活著走到終點的機率更高。

### 最佳走法（策略）

| (列/行) | 0 | 1 | 2 | 3 | 箭頭示意 |
| :---: | :---: | :---: | :---: | :---: | :---: |
| **0** | 0 (左) | 3 (上) | 3 (上) | 3 (上) | ← ↑ ↑ ↑ |
| **1** | 0 (左) | 0 (左) | 0 (左) | 0 (左) | ← ← ← ← |
| **2** | 3 (上) | 1 (下) | 0 (左) | 0 (左) | ↑ ↓ ← ← |
| **3** | 0 (左) | 2 (右) | 1 (下) | 0 (左) | ← → ↓ ← |

- **動作編號：** 0=左、1=下、2=右、3=上
- 大多數格子往左或上走——雖然終點在右下角，但冰面滑溜，貼著牆邊走、遠離洞口才是最安全的路線
- (3,2) 選「↓」：已在終點正上方，直接往下就到了
- (3,1) 選「→」：往右一格再往下就是終點

### 為什麼成功率只有 74%？

因為冰面是滑的，就算每次都選最好的方向，還是可能滑進洞裡。74% 已經是在這個滑溜環境中能達到的理論上限——沒有任何走法能做得比這更好。

## 8. VI vs PI 比較

| 特性 | 價值疊代 (VI) | 策略疊代 (PI) |
|------|--------------|--------------|
| **過程中有沒有走法** | ❌ 沒有，只有分數表 | ✅ 有，`policy[s]` 一直存在 |
| **更新分數的方式** | 試 4 個方向，取最高分 | 只算走法指定的那個方向 |
| **走法怎麼來** | 最後從分數推導 | 每輪主動更新 |
| **收斂輪數（本例）** | **324 次遍歷** | **7 次改進** |
| **每輪計算量** | 低（一次快速更新） | 高（評估要跑到內層收斂） |
| **最終走法** | 完全相同 | 完全相同 |
| **最終成功率** | 74.00% | 73.80% |

### 一句話比較

**VI**：「先把整張地圖的分數標清楚，最後一次決定走法。」每次只微調一點點，需要很多輪。

**PI**：「先定走法，徹底算清楚這套走法的分數，再大幅改進走法。」每次改進幅度大，輪數少很多。

兩者最終結果相同，差別只在「怎麼到達那個答案」。

## 9. 侷限性與為何需要下一個方法

DP 能精確算出最佳走法，但有兩個問題讓它在真實世界裡沒辦法用。

### 問題一：必須知道所有規則

```python
env.P[s][a] = [(機率, 下一格, 獎勵, 是否結束), ...]
```

FrozenLake 的規則可以查表，所以 DP 能用。但現實世界呢？

- 下圍棋：每一步之後的局面機率是多少？→ 根本算不出來
- 控制機械手臂：物理碰撞的精確機率？→ 沒有公式
- 股票交易：明天漲跌的機率？→ 沒人知道

→ **這催生了「不需要知道規則」的方法：蒙特卡羅（MC）直接讓 Agent 去玩，從實際結果裡學。**

### 問題二：格子數一多，表格就爆了

本例 16 個格子 × 4 個方向 = 64 格，小事一樁。

但如果是 Atari 電玩遊戲，畫面有 84×84 個畫素，每個畫素有 256 種顏色——狀態數量是天文數字，根本存不下這張表。

→ **這催生了深度強化學習（DQN），用神經網路取代表格，不管狀態有多複雜都能處理。**

### 演化脈絡

```
DP（1950s）
  「知道所有規則 → 直接精確計算最佳走法」
  問題：真實世界規則未知
    ↓
蒙特卡羅（1980s）
  「不知道規則 → 實際去玩，從結果學習」
  問題：必須等一整局結束才能更新
    ↓
Q-Learning / SARSA（1989/1994）
  「每走一步就更新，不用等到結束」
  問題：狀態太多，表格存不下
    ↓
DQN（2013）
  「用神經網路取代表格，處理高維輸入」
```

## 10. 結語

### 這個章節學到了什麼

| 概念 | 白話說明 |
|------|---------|
| **MDP 模型** | 一份完整的「遊戲規則表」，記錄每個格子每個方向會去哪裡、得幾分 |
| **V(s) 格子分數** | 站在這個格子，按照某套走法繼續走，預期能得幾分 |
| **Q(s,a) 方向分數** | 在這個格子選擇某個方向，預期能得幾分 |
| **貝爾曼方程式** | 一個格子的分數 = 走出去的獎勵 + 下一格的分數（打折後）|
| **折扣因子 γ** | 越遠的獎勵越不值錢，γ=0.99 代表每走一步折扣 1% |
| **價值疊代 (VI)** | 反覆標分數，直到所有格子的分數不再變，最後推出走法 |
| **策略疊代 (PI)** | 先定走法 → 算分數 → 改走法 → 重複，直到走法不再改變 |

### 最重要的一件事

DP 證明瞭：**只要知道完整規則，最佳走法一定可以被算出來。**

這個「用相鄰格子的分數來更新自己的分數」的核心想法，貫穿了整個強化學習的發展。後面所有方法，不管是 MC、Q-Learning 還是 DQN，核心都是這個想法的延伸——只是放寬了「必須知道規則」和「格子數量有限」這兩個限制。
